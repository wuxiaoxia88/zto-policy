## 1. 项目背景与目标

### 1.1 背景
中通"新面单返利政策(浮动基数模式)"结构复杂:12 个档位、两项叠加补贴、只作用于超基数部分,且各管区/时间段的基数比例与分层补贴存在差异。网点经营者普遍难以准确理解政策影响,更难以基于自家数据做出合适的业务策略选择。

### 1.2 目标用户
- **主要用户**:网点老板、店长(经营决策者)
- **次要用户**:网点财务、运营主管(使用 Excel 做精细测算)
- **传播对象**:朋友圈/微信群/管区领导(长图报告)

### 1.3 核心价值主张
> 让网点老板在 3 秒内看到"新政策能多拿多少钱",在 3 分钟内完成自家业务的"要不要冲量、要不要接边际客户"的决策模拟。

### 1.4 非目标(明确不做)
- 不做完整 P&L 建模(该政策与发件成本构成无直接耦合)
- 不建模面单成本、中转费、派费等变动项(政策不影响它们)
- 不涉及派件端任何数字(该政策只针对出港/发件)

## 2. 产品定位与交付物

### 2.1 三个交付物,一套数据模型

| 交付物 | 形态 | 核心受众 | 核心价值 |
|---|---|---|---|
| **A. Web 模拟器** | 单 HTML 文件,浏览器打开即用(无需服务器) | 网点老板、店长 | 交互式决策:拖滑块看利润曲线、对比多方案 |
| **B. Excel 工具** | 单 .xlsx 文件,纯公式零宏 | 网点财务、运营主管 | 透明可审计:每个数字都能追溯公式 |
| **C. 长图报告** | 单张 PNG,约 1080 × 3000px | 朋友圈/微信群转发对象 | 一张图讲完"政策红利 + 该怎么干" |

### 2.2 共享核心
三者共用同一套数学模型:
- 政策公式表(12 档位、基数比例、分层补贴)——**默认值 + 用户可覆盖**
- 策略模拟算法(返利 × 规模曲线、盈亏临界点)

**开发顺序**:先把"核心数据模型 + 政策公式"沉淀为单一事实源(JS 对象 / JSON),Web 和 Excel 都引用同一份逻辑,长图由 Web 端一键生成,避免三个交付物数字不一致。

### 2.3 部署形态
- **Phase 1**:纯单文件(Web 为单 HTML,Excel 为单 .xlsx,长图为动态生成 PNG),零服务器依赖
- **Phase 2 预留**:如果需要云端多网点数据管理、多用户协同,再基于设计文档另立项目

## 3. 数据模型与计算引擎

### 3.1 政策档位默认值(江苏管区)

| 档位 | 发件量区间(日均) | 基数比例 | 分层补贴(元/票) |
|---|---|---|---|
| 1  | [0, 500]         | 100% | 0.98 |
| 2  | [501, 1000]      | 80%  | 0.78 |
| 3  | [1001, 3000]     | 70%  | 0.68 |
| 4  | [3001, 5000]     | 60%  | 0.58 |
| 5  | [5001, 10000]    | 50%  | 0.48 |
| 6  | [10001, 20000]   | 40%  | 0.38 |
| 7  | [20001, 30000]   | 30%  | 0.28 |
| 8  | [30001, 40000]   | 20%  | 0.18 |
| 9  | [40001, 50000]   | 15%  | 0.13 |
| 10 | [50001, 80000]   | 10%  | 0.08 |
| 11 | [80001, 130000]  | 5%   | 0.03 |
| 12 | [130001, 999999] | 2%   | 0    |

**政策1(超基数补贴)**:1.2 元/票(作用于超基数票量)

### 3.2 政策参数可配置化(重要)

**原则**:不同时间段、不同管理区的基数比例和分层补贴存在差异,必须允许用户覆盖默认值。

- **Web 模拟器**:精细模式下展开"政策参数"面板,12 档 × 2 列(基数比例、分层补贴)全部可修改;右侧常驻"恢复默认值"按钮;改动后全局实时刷新
- **Excel 工具**:"政策档位表" sheet 中,基数比例和分层补贴列用**蓝色字体**(表示可改);档位区间列保持黑色只读
- **预设文件**:`policy-presets.json` 预置多个管区 × 时间段的政策参数,用户可下拉切换

### 3.3 关键参数与两套"超基数"(核心认知)

**⚠️ 政策1 和 政策2 的"超基数"是两个不同概念**:

| 参数 | 含义 | 变化频率 |
|------|------|---------|
| **Q** | 日均发件量 | 每月不同 |
| **r** | 档位基数比例 | 档位跳变时变化 |
| **s** | 档位分层补贴 | 档位跳变时变化 |
| **B** | **管区固定基数**(省区给每个网点单独制定) | **一般不变** |

- 政策1 超基数 = `Q × (1 − r)`  ← 按档位比例
- 政策2 超基数 = `max(Q − B, 0)`  ← 按管区固定基数

**结算方式**:日结月回算,按整月日均 Q̄ 确认档位,月度返利 = 日均单日返利 × 当月天数。

### 3.4 核心单一事实源 `model.js`

```js
const DEFAULT_TIERS = [
  { id:1,  range:[0,500],       baseRatio:1.00, tierSubsidy:0.98 },
  { id:2,  range:[501,1000],    baseRatio:0.80, tierSubsidy:0.78 },
  { id:3,  range:[1001,3000],   baseRatio:0.70, tierSubsidy:0.68 },
  { id:4,  range:[3001,5000],   baseRatio:0.60, tierSubsidy:0.58 },
  { id:5,  range:[5001,10000],  baseRatio:0.50, tierSubsidy:0.48 },
  { id:6,  range:[10001,20000], baseRatio:0.40, tierSubsidy:0.38 },
  { id:7,  range:[20001,30000], baseRatio:0.30, tierSubsidy:0.28 },
  { id:8,  range:[30001,40000], baseRatio:0.20, tierSubsidy:0.18 },
  { id:9,  range:[40001,50000], baseRatio:0.15, tierSubsidy:0.13 },
  { id:10, range:[50001,80000], baseRatio:0.10, tierSubsidy:0.08 },
  { id:11, range:[80001,130000],baseRatio:0.05, tierSubsidy:0.03 },
  { id:12, range:[130001,999999],baseRatio:0.02,tierSubsidy:0    }
];
const POLICY1 = 1.2;

function findTier(q, tiers) {
  return tiers.find(t => q >= t.range[0] && q <= t.range[1]);
}

// regionalBase(B):管区固定基数,默认 0(等价于无管区基数约束,政策2 = Q × (1-r) × s 简化情形)
// 当 B > 0 时,政策2 按 max(Q - B, 0) × s 计算
function dailyRebate(dailyQty, regionalBase = 0, tiers = currentTiers) {
  const t = findTier(dailyQty, tiers);
  // 政策1:按档位比例
  const p1ExcessQty = dailyQty * (1 - t.baseRatio);
  const policy1 = p1ExcessQty * POLICY1;
  // 政策2:按管区固定基数
  const p2ExcessQty = Math.max(dailyQty - regionalBase, 0);
  const policy2 = p2ExcessQty * t.tierSubsidy;
  return {
    tier: t.id,
    p1ExcessQty,
    p2ExcessQty,
    policy1,
    policy2,
    total: policy1 + policy2,
    perTicketActual: dailyQty > 0 ? (policy1 + policy2) / dailyQty : 0
  };
}

function monthlyRebate(avgDaily, days = 30, regionalBase = 0, tiers = currentTiers) {
  const d = dailyRebate(avgDaily, regionalBase, tiers);
  return { ...d, days, monthly: d.total * days };
}
```

### 3.5 用户输入字段

| 字段 | 含义 | 默认值 | 举例(金阊一部) |
|---|---|---|---|
| 日均发件量 Q | 判档位、算返利的必填输入 | - | 8,113 票 |
| 当月天数 | 可改 28-31 | 30 | 31 |
| **管区固定基数 B** | **省区制定的网点专属基数,政策2 的起算线** | **0**(不约束) | 按实际填,如 4,500 |
| 月度固定成本(发件端) | 1-5 行分项,用于 Layer 2 | 0 | 约 4 万元 |
| 半固定成本阶梯规则 | 用户自定义("每增 N 票日均加 ¥M 成本") | 空 | 可跳过 |
| 发件端占比 | 公共成本分摊用,按近 3 月收派比例手动填 | 33% | 按实际填 |

> 注:已**移除旧政策月返利**字段和所有新旧对比功能,工具只聚焦新政策本身的测算。

### 3.6 输出指标

**Layer 1 - 返利可视化**:
- 月度总返利、单票返利、当前档位
- **政策1 收益 / 政策2 收益独立拆分展示**
- 政策1 超基数票量、政策2 超基数票量(两者可能不同)
- 返利-发件量曲线(**动态范围:当前档 ±2 档**,边界档位截断)

**Layer 2 - 决策边界**:
- "返利覆盖固定成本"临界点:日均发多少票,返利就足以 cover 整月固定成本
- 单票视角:单票返利 vs 单票固定成本(随量下降曲线),两线交点就是"发件端自给自足点"
- 规模摊薄收益:发件量从 X 升到 Y,固定成本从 ¥A/票 → ¥B/票,叠加返利变化,每月净额外收益 +¥Z

### 3.7 明确的边界
- ❌ 不做完整 P&L
- ❌ 不建模面单成本、中转费、派费等变动项
- ❌ 不涉及派件端
- ❌ 不做跨档日波动(简易模式按整月日均值回算)
- ❌ 不做新旧政策对比(已于 2026-04-23 移除)

**视觉规则**:

### 4.3 滑块模拟器

- **左**:日均发件量滑块(100 ~ 80,000 票,对数刻度)
- **中**:最醒目的月度返利大数字(72px 金色),随滑块实时更新
- **右**:4 张数据卡,每张一个大数字:
  - 当前档位(大号徽章)
  - 实际单票返利(48px 金色)
  - 政策1 超基数票量 / 日(48px)

- **对比看板 · Pin 机制**:点击"📌 钉住当前方案" → 下方生成小卡,最多钉 4 个方案横向对比(方案快照包含 B 值)

### 4.4 Layer 2 决策 Tab

**Tab A - 月度视角**:
- 横轴:日均发件量(当前档 ±2 档范围)
- 纵轴:月度金额
- 两条线:月度返利曲线(上升)+ 固定成本(水平线)
- 关键数字气泡:两线交点(返利 cover 固定成本的临界发件量)

**Tab B - 单票视角**:
- 横轴:日均发件量(当前档 ±2 档范围)
- 纵轴:元/票
- 两条线:单票返利(档位阶跃曲线)+ 单票固定成本(下降曲线)
- 关键数字气泡:交点(单票"自给自足"点)

### 4.5 简易模式 vs 精细模式

**简易模式(默认)**:
- 必填:日均发件量
- 其他默认:管区固定基数 B = 0(退化为按档位比例算政策2),固定成本 = 0
- Hero 区只显示返利本身,不拆政策1/政策2

**精细模式**:
- 展开所有输入:**管区固定基数 B**、月度固定成本(1-5 行)、阶梯规则、发件占比
- **政策参数面板可编辑**(12 档 × 2 列)
- Hero 区拆分展示政策1 / 政策2 收益
- Layer 2 决策区完全可用

### 4.6 数字至上的视觉规范
- 图表标题、轴标签用灰色小字,数字标注用粗体大号
- 关键节点(档位跳变、盈亏平衡点)用**数字气泡**标注,不用文字
- 颜色语义:**中通蓝(#0B4EA2)**=政策/档位(主色调)、金色=返利/红利、绿色=增长、灰色=成本
- 其他颜色细节后期调整

### 6.2 版式(自上而下 8 段)

```
① 顶部横幅(中通蓝背景)
```

## 10. 附录:关键设计决策记录

本附录记录 brainstorming 过程中的重要决策与其理由,便于未来回溯或迭代时参考。

### 10.1 形式选择:Web 为主 + Excel + 长图(Q1)
- **决策**:A(网页模拟器)主体 + C(图文报告)导出功能
- **理由**:政策核心难点(基数比例随档位变化、超基数补贴的非线性)必须靠交互式曲线和滑块才能让人秒懂,PPT 和 Excel 静态形式讲不清楚;但 Excel 对财务人员仍有价值,长图对传播有价值
- **后续追加**:用户明确要求 Excel + Web 双版本

### 10.2 财务颗粒度:从完整 P&L 收敛到"返利独立"(Q2 + 用户补充)
- **最初设计**:完整 P&L(收入、固定/半固定/变动成本、返利、净利润)
- **用户澄清**:返利政策独立于发件成本,政策只影响返利,不影响面单成本、中转费、派费等
- **最终决策**:工具只管返利本身 + 返利对决策边界的影响(Layer 2),不算完整 P&L
- **影响**:大幅简化了数据模型,更聚焦

### 10.3 交互流程:D 组合(滑块为主 + 对比看板)(Q3)
- **决策**:D = B(滑块模拟型) + C(策略对比型) 组合
- **理由**:滑块看曲线秒懂政策;对比看板支持横向方案比较;两者互补形成决策闭环
- **额外**:用户要求"简易模式"(原"小白模式"改名),降低入门门槛

### 10.4 Hero 视觉锚点:A 红利对比(Q4)
- **决策**:A(红利对比) + 基数线为基准(当时备选 Q5)
- **理由**:"多赚了多少"比"别亏"的获得感更适合中国网点老板;"超基数奖励"直接契合政策逻辑,零外部数据依赖
- **对比基准**:无同档中位数数据,采用"基数线对比"(Q5 最终选 A),旧政策对比作为可选开关(需要用户录入参数)

### 10.5 Layer 2 玩法:A+B 起步,C 推迟(Q6)
- **决策**:A(边际成本曲线)+ B(客户分层模拟器)组合,C(跨期博弈)放 Phase 2
- **理由**:A 是地基,B 回答老板最高频的决策问题,C 涉及时间维度假设爆炸
- **Phase 1 收敛**:实际只做 Layer 2-A(规模曲线),B 也推迟到 Phase 2

### 10.6 档位判断:按日均,不按月度(用户修正)
- **初始错误**:按月度发件量判档位(把金阊一部 25 万票算成第 12 档)
- **用户修正**:档位按日均,不能用整月数据算档位
- **修正后**:金阊一部日均 8,113 → 第 5 档,月返利 ¥211,263
- **简易模式**:假设每天 = 日均,整月用同一档位;精细模式按天(Phase 2)

### 10.7 政策参数可配置化(用户补充)
- **用户补充**:不同时间段、不同管理区的基数比例和分层补贴有差异
- **决策**:政策档位表从"锁定单一事实源"改为"默认值 + 用户可覆盖"
- **实现**:Web 精细模式 + Excel 蓝字可编辑单元格 + `policy-presets.json` 多预设

### 10.8 曲线图范围:当前档 ±2 档(用户优化)
- **初始设计**:曲线图覆盖 0 ~ 1,000,000 票全量
- **用户优化**:现实中不存在跨度 ≥10 档的决策场景,曲线只需当前档 ±2 档
- **边界处理**:第 1/2 档左侧不够 2 档时自动截断;第 11/12 档右侧同理
- **影响**:曲线更清晰,Excel 的"曲线数据"sheet 行数大幅缩减

### 10.9 Hero 数字动画:不做(用户决定)
- **初始提议**:0.8 秒数字滚动开奖动画
- **用户决定**:直接用醒目数字,不要动画
- **理由**:数字本身已足够醒目,动画反而拖慢感知速度

### 10.10 长图含网点署名(Q6-A)
- **决策**:A 方案,Hero 和底部都显示网点名称
- **理由**:专属感强,老板愿意转发;底部附免责声明防止冒用

### 10.11 决策建议暂缓(用户决定)
- **初始提议**:基于规则自动生成 2-3 条决策建议
- **用户决定**:先预留位置,后面有完善想法时再加
- **实现**:长图第 ⑦ 段放占位文字;Web 端在 Layer 2 区域保留空白 card

### 10.12 引入管区固定基数 B 与移除新旧对比(2026-04-23 晚修订)
- **用户补充**:"有个重要的参数需要引入,即管区定给每个网点的基数"
- **核心认知修正**:
  - **政策1 超基数 = Q × (1 − r)**,按档位基数比例(与旧理解一致)
  - **政策2 超基数 = max(Q − B, 0)**,按管区固定基数(**新**,与档位比例无关)
  - 两者是独立计算,可能不相等
- **用户例子**:Q=4999 B=4500 → 政策1 = 4999×40%×1.2 = ¥2,399.52,政策2 = 499×0.58 = ¥289.42
- **另一修正**:去掉所有旧政策对比的内容,工具只聚焦新政策本身
- **结算方式明确**:日结月回算,按整月日均 Q̄ 确认档位,月度返利 = 日均单日 × 当月天数
- **影响**:
  - 政策源文件 §3/§4/§5 全面重写(加入参数 B 和新案例 B1/B2)
  - 设计文档 §3.3-§3.7 重构;§4.2/§4.3/§4.5/§5.3 更新;§6.2 长图版式去新旧对比,改为政策1/政策2 拆分饼图;§9 QA 用例重组为 A/B 两个系列
  - Web 实现:`dailyRebate` 增加 `regionalBase` 参数;UI 精细模式新增 B 输入;去掉旧政策月返利输入、新旧差额卡、Hero 新旧对比行
