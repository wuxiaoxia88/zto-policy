# 技术框架、UI方向与开发规划

**日期**: 2026-05-07
**项目**: 中通面单返利测算工具
**结论**: MVP 采用“本地优先、静态交付、模块化开发、单文件/静态包发布”的架构。

---

## 1. 技术框架先定

### 1.1 现有方式评估

当前项目是单个 `index.html`，优点很明显:

1. 双击可用，传播简单。
2. 不需要服务器，不需要账号，不需要安装。
3. 很适合政策讲解、微信群传播、临时测算。
4. 对网点老板和财务人员的使用门槛最低。

但继续在单个 HTML 里叠加新需求会遇到问题:

1. 多客户叠加、成本模板、业务量处罚、图表联动会让单文件越来越难维护。
2. 计算逻辑、UI 逻辑、图表逻辑混在一起，后期容易算错。
3. 模板导入、方案保存、CSV/XLSX 解析、复杂图表需要更清晰的模块边界。
4. 后续如果要扩展 Excel 导入、长图导出、PWA、云端同步，单文件会拖慢开发。

### 1.2 推荐架构

采用“开发工程化、交付轻量化”的折中方案。

| 层级 | 推荐方案 | 原因 |
|---|---|---|
| 前端框架 | React + TypeScript + Vite | 成熟、生态稳、适合复杂状态和表单 |
| UI 状态 | Zustand 或 React Context | 方案、客户、模板状态清晰可控 |
| 计算引擎 | 纯 TypeScript 模块 | 返利/处罚/成本公式独立，可单元测试 |
| 图表 | ECharts | 中文业务图表能力强，瀑布图/折线/热力图稳定 |
| 表格导入 | SheetJS/xlsx + CSV parser | 支持成本模板、客户列表导入 |
| 图片导出 | html-to-image 或 dom-to-image + Canvas fallback | 支持结果卡、方案图、政策讲解图分享 |
| 本地存储 | localStorage 起步，IndexedDB 预留 | MVP 简单，后续支持大模板数据 |
| 日期处理 | 原生 Date 或 date-fns | 自动识别月份天数 |
| 构建交付 | Vite build + 单文件打包选项 | 保留“发一个 HTML 就能用”的传播优势 |
| 测试 | Vitest | 公式必须可验证，防止政策口径改错 |

### 1.3 交付形态

MVP 推荐两个交付版本:

1. **单文件版**
   打包成一个 `index.html`，适合微信、U盘、群文件传播。用户双击即可打开。

2. **静态目录版**
   `dist/` 目录，适合部署到内网、OSS、GitHub Pages 或本地 Web 服务。后续图表、图片、导入导出能力更稳。

优先保证单文件版可用；如果某些浏览器对本地文件导入导出有限制，再提示使用静态目录版。

### 1.4 是否需要后端

MVP 不需要后端。

原因:

1. 数据敏感，网点成本、客户报价、返利测算最好先本地保存。
2. 没有登录和多人协作需求。
3. 离线传播是核心优势。
4. 先验证公式和决策流程，比搭服务器更重要。

后端只作为第三阶段预留，用于多网点、多用户、云端模板、历史数据同步。

### 1.5 核心模块拆分

```text
src/
  core/
    rebate.ts          # 返利、档位、分层补贴
    penalty.ts         # 业务量处罚
    cost.ts            # 发件成本、重量段成本
    customer.ts        # 单客户与叠加客户测算
    increment.ts       # 存量收益/新增件亏损/增量减利分析
    recommendation.ts  # 决策建议
    date.ts            # 月份天数
  data/
    policy-presets.ts  # 苏南模板、旧江苏模板
    cost-presets.ts    # 成本模板示例
  state/
    store.ts           # 当前测算状态
  components/
    Dashboard/
    PolicyEditor/
    CustomerStack/
    CostPanel/
    ScenarioCompare/
    Charts/
  tests/
    rebate.test.ts
    penalty.test.ts
    customer.test.ts
```

### 1.6 技术决策

最终建议:

> 使用 React + TypeScript + Vite 重构，而不是继续维护单个手写 `index.html`；但发布时仍提供单文件 HTML，保留原项目最适合传播和使用的优势。

## 2. UI方向讨论

### 2.1 UI目标

UI 不是做给专业财务系统用户看的，而是给网点老板、店长、业务负责人快速决策用的。核心目标:

1. 第一眼看懂当前能拿多少返利。
2. 一拖滑块就看懂跳档变化。
3. 一加客户就看懂赚不赚、少罚多少、最低报价多少。
4. 图表能解释原因，不只是展示曲线。
5. 高级参数可以展开，但不能挡住主流程。
6. 支持管区宣传和网点经营测算两种不同使用场景。

### 2.2 双视角 UI

MVP 顶部必须提供两个视角:

1. **管区视角**
   侧重政策宣传、政策讲解、政策公平性/合理性/普惠性的展示。默认弱化成本细节，突出档位曲线、跳档收益、典型网点对比和传播图片。

2. **网点视角**
   侧重经营测算、客户开发、客户报价和成本收益判断。默认展开客户、成本、处罚、报价、毛利和决策建议。

切换视角时不清空数据，只改变默认布局、文案和导出内容。

### 2.3 推荐页面结构

MVP 先做一个工作台，不做复杂多页面。

```text
顶部工具栏
  管区视角 / 网点视角
  政策模板 / 测算月份 / 参数导入 / 保存方案 / 导出数据 / 保存图片

左侧输入区
  当前网点
  业务量指标
  政策参数
  成本简易输入

中间决策区
  核心结果卡
  日均票量滑块
  多客户叠加列表
  决策建议

右侧可视化区
  返利曲线
  处罚风险图
  瀑布图
  方案对比
```

在窄屏或投屏时改为上下布局:

```text
核心结果
滑块模拟
客户叠加
图表解释
方案对比
参数设置
导出分享
```

### 2.4 首屏应该展示什么

首屏必须直接是工具，不做宣传页。

建议首屏 6 个核心数字:

1. 月度总返利。
2. 当前档位。
3. 单票返利。
4. 距离下一档还差多少票。
5. 当前业务量处罚风险。
6. 方案后月度毛利增量。

下面直接放滑块和“追加客户”按钮。

管区视角首屏更强调:

1. 政策返利曲线。
2. 典型网点跳档示例。
3. 政策公平性/普惠性说明。
4. 一键保存讲解图。

网点视角首屏更强调:

1. 当前月返利。
2. 业务量处罚。
3. 客户叠加。
4. 保本报价和毛利增量。

### 2.5 关键交互

1. **日均票量滑块**
   拖动后实时刷新档位、返利、处罚、毛利。

2. **追加客户列表**
   每行一个客户，可以调整票量、报价、成本、退件率、启用/停用。

3. **边际贡献提示**
   每个客户旁边显示“带来返利 +X、减少罚款 +Y、毛利 +Z”。

4. **增量减利提示**
   当票量增加但利润下降时，明确提示“增量减利”，并拆解为“存量收益提升”和“新增件自身亏损”。

5. **方案 Pin**
   保存当前方案，和其他方案横向比较。

6. **缺失数据提示**
   成本没填时不阻断测算，但建议区明确提示“当前只能判断返利和处罚，不能完整判断毛利”。

7. **参数设置双入口**
   所有关键参数都提供手工输入和导入两种方式。手工输入用于现场试算，导入用于批量设置政策、客户、成本和业务量指标。

8. **导出分享**
   当前测算结果、方案对比、管区政策讲解图、网点报价测算图都要能导出图片；明细数据可导出 CSV/XLSX。

### 2.6 图表原则

1. 曲线视觉要平滑，但政策跳档点必须清楚标出来。
2. 少用专业术语，多用“多拿”“少罚”“还差”“最低报价”。
3. 图表默认只显示当前决策附近区间，不展示十万票以上全局大图。
4. 瀑布图用来解释“为什么值得做”，不要只做装饰。
5. 图表颜色固定:
   - 蓝色: 政策/返利。
   - 绿色: 收益增加。
   - 红色: 处罚/风险。
   - 灰色: 成本。
   - 金色: 最终结论或关键金额。

### 2.7 UI阶段结论

MVP UI 采用“双视角经营决策工作台”。管区视角用于政策传播，网点视角用于经营决策。第一版不追求复杂动画，先追求数据正确、结论清楚、交互顺手、能导出分享。

## 3. 开发规划

### 3.1 开发顺序

先做计算引擎，再做 UI，再做导入导出。

原因:

1. 本项目最怕公式口径错。
2. 业务量处罚和客户叠加会影响最终建议，必须先有稳定数据模型。
3. UI 可以迭代，但计算模型一开始要可测试。

### 3.2 MVP 任务拆分

#### Sprint 1: 核心计算

产出:

1. 苏南 9 档政策模板。
2. 月份天数自动识别。
3. 返利计算。
4. 业务量处罚计算。
5. 单元测试。

验收:

1. 2026-05 自动 31 天。
2. 2026-06 自动 30 天。
3. 8000 指标、7500 实际的处罚能算出 5250/6000 两种结果。
4. 苏南政策样例计算正确。

#### Sprint 2: 工作台雏形

产出:

1. 顶部政策/月选择。
2. 管区视角 / 网点视角切换。
3. 核心结果卡。
4. 日均票量滑块。
5. 基础返利曲线。
6. 参数手工输入入口。
7. 参数导入入口。

验收:

1. 用户打开页面就能输入票量并看到月返利。
2. 拖动滑块能看到跳档变化。
3. 用户能在管区视角和网点视角之间切换，且数据不丢。

#### Sprint 3: 客户叠加

产出:

1. 客户列表。
2. 新增/删除/复制客户。
3. 客户件量滑块。
4. 客户启用/停用。
5. 边际贡献计算。

验收:

1. 至少可追加 5 个客户。
2. 任意调整客户件量后，总结果实时刷新。

#### Sprint 4: 成本与建议

产出:

1. 简易成本输入。
2. 保本报价。
3. 毛利增量。
4. 决策建议文案。
5. 缺失数据提示。

验收:

1. 成本为空时仍能测返利和处罚。
2. 成本填写后能输出毛利和最低报价。
3. 建议必须说明数据依据。

#### Sprint 5: 图表与方案对比

产出:

1. 返利曲线。
2. 处罚风险图。
3. 简化瀑布图。
4. 方案保存和对比。
5. 管区视角政策讲解图。
6. 网点视角方案对比图。

验收:

1. 至少保存 3 个方案。
2. 非专业用户能看懂哪套方案更优。
3. 管区视角能生成适合传播的政策讲解图。

#### Sprint 6: 模板与导入

产出:

1. 政策模板保存。
2. 成本模板导入。
3. 客户参数导入。
4. 业务量指标导入。
5. 测算数据导出。
6. 图片保存。
7. 本地保存。
8. 单文件打包。

验收:

1. 用户刷新页面后数据不丢。
2. 可导入发件成本表。
3. 可导入客户列表和业务量指标。
4. 可导出 CSV/XLSX 数据。
5. 可保存分享图片。
6. 可生成传播版 HTML。

### 3.3 不在 MVP 做

1. 登录。
2. 云端同步。
3. 多网点权限。
4. 自动连接中通系统。
5. 复杂动画和大屏特效。
6. 完整独立成本系统。

### 3.4 文件清理原则

本目录中以下文件属于末端政策，不作为本项目核心参考:

1. `苏网字【2026】008号-末端&新产品KPI考核方案.md`

保留文件不删除，但 README 和 PRD 不再把它们列为核心资料，避免后续开发时误用末端口径。

## 4. 最终建议

1. **技术框架**: React + TypeScript + Vite + ECharts，本地优先，无后端。
2. **交付方式**: 开发工程化，发布单文件 HTML 和静态目录两种版本。
3. **UI方向**: 双视角经营决策工作台，管区视角用于政策推广，网点视角用于经营测算。
4. **参数设置**: 所有关键参数都提供手工输入和文件导入两种方式。
5. **传播能力**: MVP 就提供数据导出和图片保存。
6. **开发策略**: 敏捷迭代，先公式、双视角和客户叠加闭环，再做复杂模板和高级图表。
7. **现有原型定位**: `index.html` 作为旧版交互参考，不继续在原文件上无限堆功能。
